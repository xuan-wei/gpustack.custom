## 2. 实测结论

在「本机 server + 远程 worker」真实环境用浏览器 + 脚本压测，**双后端全链路通过**。

### llama.cpp 路径（`qwen2.5-0.5b-instruct`，`--parallel=1`）
| 行为 | 结果 | 证据 |
|------|------|------|
| 冷启动自动加载 | ✅ | 首请求 `replicas 0→1` |
| 扩容 | ✅ | `waiting=7` 持续 ~1m45s → `1→2`（`up: waiting=7.0`）|
| 上限封顶 | ✅ | 到 `auto_load_replicas=2` 停 |
| 缩容 | ✅ | 停负载 `running→0` → `2→1`（`pred_util=0.00`）|
| Auto-Drain | ✅ | victim 标 draining 后被删，无残留 |

### vLLM 路径（`qwen2.5-14b-instruct-awq`，KV 驱动）
| 行为 | 结果 | 证据 |
|------|------|------|
| 冷启动 | ✅ | `replicas 0→2`（直接拉到 `auto_load_replicas`）|
| KV 压力 | ✅ | `kv: 0→0.54→0.98→0.99` 打满，`waiting: 0→46` |
| 扩容 | ✅ | `1→2`（`up: waiting=46.0, kv_sum=0.34`）|
| 缩容（KV 路径） | ✅ | KV 回落到 0 → `2→1`（`down: pred_kv=0.00, victim=…(kv=0.00)`）|

### 独立验证
- **Auto-Unload** ✅：0.5b 设 `timeout=1min`，空闲后 `replicas→0`。
- **生命周期字段 PUT 持久化** ✅：GET-改-PUT-再GET 一致。
- **手动跨节点 GPU**（前几轮已测）✅。

### 两条路径的差异（重要）
- **llama.cpp**：`waiting` = 并发超过 `--parallel` 立即产生（`requests_deferred`）；缩容判据 `running_total/((replicas−1)×slot_capacity)`。
- **vLLM**：`waiting` = KV cache 打满后 sequence 排队；缩容判据 `kv_sum/(replicas−1)`。

### 本轮修掉的 3 个问题
1. **`auto_load_replicas` 默认 1 时扩容静默不生效**（它同时是冷启动数和扩容上限 `ceiling=max(1,auto_load_replicas)`）。**不拆字段**（本就最小为 1），改为 i18n tooltip 讲清耦合：`autoAdjustReplicas.tips` 说明「上限=自动加载副本数、下限=1，需扩容请设 >1」。
2. **改 `backend_parameters` 不重建实例 → `load_ratio` 短暂错配**（曾显示 800%）。确认 `ModelInstance` 不持久化 resolved args，无法从实例解析 slot_capacity；改为 `get_runtime_snapshots` 把 `load_ratio` **clamp 到 [0,1]**（利用率本就 0–1，>100% 无意义，且会随重建自愈）。
3. **小显示器上实时负载列右侧刷新按钮被裁掉**。把刷新按钮从单元格第 2 行移到**列表头**（永远可见；功能上本就是全局 scrape）。

## 4. 修改思路

贯穿所有改动的指导原则（也是 rebase 成本最低原则，见 [CLAUDE.md](../CLAUDE.md)）：

1. **优先新增文件**。所有重逻辑（伸缩、卸载、drain、聚合、冷启动 gate、显存换算）都落在**新文件**里 → 零 git 冲突。官方文件只留「装配点」和「最小钩子」。
2. **改官方文件时只做最小钩子**：`server.py` 只加任务启动调用；`schemas/models.py` 只加字段；`metrics_collector.py` 只贴一行同类字段。不顺手重构。
3. **生命周期状态进 DB**：16 个字段 + `is_draining` 用单个 consolidated 迁移；新 datetime 字段用 `UTCDateTime()`（否则丢 session-TZ 偏移）。
4. **后台任务解耦**：5 个任务（auto_metrics 30s / auto_unload 30s / runtime_aggregator 30s / auto_scaling 30–60s / auto_drain 10s）各自独立，通过 ORM + `app.state.runtime_metrics_aggregator` 通信，不互相硬依赖。
5. **缩容用「标记 draining + 延迟删」而非直接删**：消除缩容时网关端点未收敛导致的 404 竞态；`is_draining` 在同一事务里和 `replicas--` 一起提交，避免 `sync_replicas` 观察到中间态生成幽灵副本。
6. **跨后端统一指标**：worker `/metrics` 的 `llamacpp:*`/`vllm:*` 经 `metrics_config.yaml` 统一成 `gpustack:*`，聚合器只认 `gpustack:*`；`load_ratio` 把 vLLM 的 KV 占用和 llama.cpp 的 slot 占用归一到 0–1。
7. **手动选 GPU 尊重用户的 distributed=false**：官方多处「跨 worker 选 GPU ⟹ 强制 distributed」的隐式逻辑被解除（详见 §5）。
8. **通用 bug 在 fork 里打 patch，原则上不上游**（除非高频）；但已标注哪些「宜上游 PR」以便日后收敛 fork 分歧。

## 5. 跨文件耦合修复

「**手动选 GPU + 取消跨节点推理**」横跨 3 个后端文件，rebase 时**必须作为整体校验**，只合一半会产生不一致（如校验放行但调度器又强制 distributed）：

1. `scheduler/scheduler.py` —— 删除 `evaluate_gguf_model` 里强制 `distributed_inference_across_workers=True` 的块。
2. `routes/models.py::validate_gpu_ids` —— custom-backend 跨 worker 拒绝条件加 `and distributed_inference_across_workers`。
3. `gguf_resource_fit_selector.py::_should_skip_for_manual_scheduling` —— `worker_num>1` 强制 multi-worker finder 时加 `and distributed_inference_across_workers` 守卫；并把 pre-check 由「任一 GPU 不足即失败」改「全部不足才失败」（否则多副本铺开时，一张被占的 GPU 会拖垮整批候选）。

配套前端：`forms/index.tsx`（initialValues 默认 `false`）+ `forms/schedule-type.tsx`（切手动时 `setFieldValue false`）。

> ⚠️ 自动 merge 最危险的点：守卫是给既有 `if` 追加 `and ...` 条件，自动 merge 可能**静默丢掉**这个 and。rebase 后必须人工确认三处守卫都在、删除块没复活。

## 8. 其他重要维度

### 部署层（`gpustack.custom/`，完全自有，零 rebase 风险）
- `Dockerfile.custom`：multi-stage 构建（`base` → `server`/`worker`）。`base` 叠加后端源码 + grafana 面板 + entrypoint；`server` 额外叠加前端 dist；`worker` 纯后端，不需要编译前端。
- `docker-compose-server.yaml` / `docker-compose-worker.yaml` + `.env-server-dev` / `.env-worker-dev`。
- ModelScope 缓存由 `entrypoint-custom.sh` 软链，**不改** `config/config.py`/`cmd/start.py`。
- Grafana 自定义 row（`gpustack/docker-compose/grafana/grafana_dashboards/gpustack-model.json` +533 行，在 gpustack 子模块内）：给 llama.cpp 补网关侧请求/延迟/吞吐面板。

### 新增环境变量（`envs/__init__.py`）
- `PROXY_CONNECT_TIMEOUT`（5s）、`PROXY_SOCK_READ_TIMEOUT`（60s）：代理超时硬化。
- `COLD_START_MAX_WAITERS`（300）：单模型冷启动并发 waiter 上限，防 handler/DB 池耗尽。

### 关键参数语义（易踩坑）
- `auto_load_replicas`：**冷启动副本数 + 伸缩上限**，二义；最小 1。要扩容须设 >1。
- `scale_window_minutes`：扩缩条件须持续满足该时长；冷却 = 窗口/2；最小 1 分钟。
- `scale_down_kv_threshold`：缩容判据阈值（vLLM 比 KV，llama.cpp 比 slot 占用），默认 0.6。
- `auto_unload_timeout`：空闲多少分钟卸载到 0（分钟）。

### 已知限制 / 设计取舍
- `load_ratio` 在「改 `backend_parameters` 但未重建实例」窗口期会偏（已 clamp 到 ≤100%，随重建自愈）。
- llama.cpp 的 TTFT 经 counter-pair 兜底时是 ms-per-prompt-token 近似，非真 TTFT。
- `token.py` 冷启动长轮询**未**接 `cold_start_gate`（仅 openai.py 路径受保护）—— 见 §7.2。
- 缩容下限恒为 1（0 由 auto_unload 负责）。
